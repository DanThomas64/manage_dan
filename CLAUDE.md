# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
cargo build                    # Debug build
cargo build --release          # Release build
cargo build -p <crate>         # Build specific crate
cargo check                    # Type-check without building
cargo test                     # Run all tests
cargo test -p <crate>          # Test specific crate
cargo test <test_name>         # Run a single test by name
cargo clippy                   # Lint
cargo fmt                      # Format code
cargo fmt -- --check           # Check formatting only
cargo run -p app                # Run the backend server (http://0.0.0.0:8080)
cargo run -p tui                # Run the TUI (needs the server running; reads MANAGE_API_URL, default http://127.0.0.1:8080)
./deploy.sh                     # Build release + install as a native systemd service + nginx reverse proxy
./deploy-frontend.sh            # Copy frontend/index.html to the nginx static path only — no build, no service restart
```

Local config lives in `config/local.toml` (gitignored, layered over `config/default.toml`); any setting can also be overridden via an `APP_`-prefixed env var (e.g. `APP_PRINTER_MODE=usb`).

Each crate versions independently — every crate's own `Cargo.toml` sets its own `version`, there is no shared workspace version. Bump only the version(s) of the crate(s) whose code actually changed in a given PR.

## Architecture

This is a Rust Cargo workspace with one binary (`app`) and library crates: `db`, `log`, `printer`, `todo`, `vikunja`, `lists`, `notes`, `project`, plus a second binary `tui`. The `app` crate depends on all library crates. `frontend/` (vanilla JS SPA, served by nginx) and `android/` are separate, non-Cargo clients of the HTTP API.

**Execution flow**: `main()` (`app/src/main.rs`) loads `AppConfig`, calls `SystemsStatus::init()` (`app/src/nogo.rs`) which sequentially calls `init()` on each subsystem crate in dependency order (db → log → notes → project → printer → todo → lists), tracking per-subsystem `Status` (Go/Nogo/Degraded/Unknown/Init). DB is initialized first so the log table exists before logging starts. `notes::init()` checks for the `nb` CLI binary; the notes subsystem goes Nogo if `nb` is not installed. `SystemsGoNogo::calculate_initial_status()` derives overall health from the fold of all subsystem statuses; `start_monitoring()` spawns a tokio task that recomputes it every 500ms. After init, the app prints a startup receipt to the configured printer (todo/list stats fetched concurrently via `tokio::join!`, with failures rendered as placeholder text rather than aborting startup), spawns background tasks (Vikunja print monitor, daily summary, end-of-day completed summary — each printed once immediately if not already run today, then rescheduled to their configured hour), then starts the `warp` HTTP API server (`app/src/api.rs`) at `/api/v1/...`.

**Standardized module pattern** — every library crate follows the same structure:
- `lib.rs`: `init() -> Result<(), Error>` and tests
- `{crate}_error.rs`: custom error type via `thiserror`
- `{crate}_prelude.rs`: re-exports for ergonomic imports

`app/src/error.rs` defines `AppError`, which wraps every subsystem's error type (`#[from] DbLibError`, `NotesLibError`, etc.) via `thiserror`'s `#[error(transparent)]`, so subsystem errors surface at the top level without manual conversion.

**Shared workspace dependencies** (declared in root `Cargo.toml`, inherited with `workspace = true`): `tokio` (full, async runtime), `tracing` (structured logging), `thiserror` (custom errors), `anyhow` (error context), `chrono` (date/time), `serde`, `config` (layered TOML + env config), `escpos` (thermal printer), `reqwest` (Vikunja HTTP client), `uuid`.

**Logging**: `tracing-subscriber` configured in `app/src/main.rs` with RFC 3339 timestamps, line numbers, ANSI disabled, DEBUG level.

**Todo subsystem**: pluggable backend selected via `[todo] backend = "vikunja" | "nb"` in config (`todo/src/backends/{vikunja,nb}.rs`, dispatched from `todo::create_item`/`read_items`/etc. in `todo/src/lib.rs`). `"vikunja"` stores tasks in a self-hosted Vikunja instance over its HTTP API (`vikunja` crate); `"nb"` stores them as files in an `nb` notebook (name set by `todo.nb_notebook`, default `"todo"`) — the same `nb` CLI the notes subsystem shells out to, with priority encoded as a `<!-- priority: N -->` HTML-comment header since `nb` has no native priority field. Priority is 0–5 (0=Unset, 1=Low, 2=Medium, 3=High, 4=Urgent, 5=Do Now), matching Vikunja's own scale — both backends and both clients (TUI, frontend) clamp/validate to this range. New todos are printed once on creation (`print_ticket_on_creation`) and, independently, by the background print monitor (`todo/src/monitor.rs`) whenever it sees changed content; because backend creation (especially `nb`, which shells out multiple times) can take long enough for the monitor's own poll to observe the same brand-new item first, both paths claim the print atomically via `db::printed_claim` (an `INSERT ... ON CONFLICT ... WHERE` that only one caller can win) before printing, rolling the claim back on print failure so the monitor retries next poll. Marking a todo *completed* (not un-completing one, and not re-completing an already-completed one — `complete_item` fetches the item first to check its prior state) also logs a daily-log entry via `notes::create_log` (`todo::log_completion`), tagged `todo-complete` and, if the todo belongs to a project, also `project-<slug>`; this is best-effort (a logging failure only warns, it doesn't fail the completion) and works under either todo backend, since the `log` notebook is a `notes`-crate concept independent of which backend stores the todo itself.

**Notes subsystem**: notes are not stored in the app's own DB — they're markdown files managed by the external [`nb`](https://xwmx.github.io/nb/) CLI, with full-text search shelled out to `nb search`. If files are edited outside `nb`, its index must be resynced with `nb index reconcile`. `nb_client.rs` always addresses notebooks with an explicit `notebook:cmd` prefix, including `home` — `nb` persists whichever notebook a colon-prefixed command last targeted as its own "current" notebook, so a bare/unprefixed command silently drifts onto that instead of `home`.

**Log feature**: a dedicated `log` notebook, driven by `nb daily`, backs `POST`/`GET /api/v1/notes/daily`. Each entry is appended to that day's file under an auto-generated `## HH:MM:SS` heading, body-formatted with the same title/tags/content layout regular notes use (`nb_client::format_note_body`). `notes::recent_logs()` re-parses those multi-entry day-files back into individual entries for the last N days. The `log` notebook is excluded from the general notes browsing surfaces (`folders`/`list`/`search`) so daily-log files don't clutter regular note browsing.

**Project subsystem** (`project` crate, `/api/v1/project...`): lets todos/notes/log-entries/lists be grouped under a named project, without changing how anything behaves when created *without* one — association is purely additive (a shared tag/folder/list-group), not a new storage layer. `project::init(base_dir)` (config: `[project] base_dir`, default `~/projects`, `~` expanded manually) creates the `projects` table (`id`, `name`, `slug`, `tag`, `list_group_id`, `fs_path`, `archived_at`, `created_at`); `create_project(name)` derives a snake_case `slug` (words joined with `_`, not `-`), creates a dedicated `lists` group (same name) with a default "General" category, an nb notebook named `slug`, a filesystem folder under `base_dir`, and a `project-<slug>` tag (the `"project-"` prefix itself stays hyphenated — it's a fixed literal, not slug-derived). Scoping reuses existing mechanisms unmodified: the `nb` todo backend already maps `TodoItem.project_title` 1:1 onto an nb subfolder (`todo/src/backends/nb.rs`), so project todos are created with `project_title = Some(slug)` and read back with `todo::read_items()` filtered by that same slug; notes/log entries are tagged `project-<slug>` and filtered via `notes::list`/`notes::recent_logs_tagged`; lists are just the project's own group, fetched via `lists::list_categories(list_group_id)`. `project_detail(id)` aggregates all four via `tokio::join!`, with each fetch best-effort (a subsystem being down blanks out just its own section, not the whole page). Deleting a project archives it, never destroys data: tagged notes and nb todo files are moved (`nb move`, which requires the destination notebook to already exist — `notes::ensure_archive_notebook()` bootstraps the shared `archive` notebook first) into `archive:<slug>/...` and `archive:<slug>/todo/` (trailing slash required, or a second todo silently overwrites the first), the project's filesystem folder is zipped (`zip` CLI) to `<base_dir>/.archive/<slug>.zip` and the live folder removed, and `archived_at` is set — the `projects` row and its `lists` group are never deleted. Log entries tagged with the project are deliberately left untouched in the shared `log` notebook (no per-entry removal is possible; daily-log files pack many entries into one shared per-day file). Vikunja-backend todos are out of scope for project association (Vikunja's `create_item` always posts to the client's single fixed `project_id` and ignores `project_title` for placement) — an accepted gap, not one this subsystem works around.

`restore_project(id)` reverses `archive_project`: notes are nested one level deep at `archive:<slug>/` once archived (`nb`'s own root-level listing doesn't descend into subfolders, and a folder-nested item is addressed as `folder/id`, not a bare id — `notes::restore_archived_notes` parses the folder-scoped listing and moves each note back to the project's own notebook root in one pass rather than going through the bare-id-only `nb_show`/`nb_move` path), todos are moved back by folder (`todo::restore_project_todos`), and the zip is unzipped back into place (`unzip -o -d <base_dir>`) and removed. `delete_project(id)` permanently destroys an already-archived project — refuses on a live (non-archived) one — by deleting the whole `archive:<slug>/` folder in one call (recursively removes the nested `archive:<slug>/todo/` too), the project's own now-empty notebook, the archive zip, the `lists` group, and finally the `projects` row itself (in that order — the row must go *before* the group, since `list_group_id` is a foreign key into it). Both endpoints are exposed as `POST /api/v1/project/:id/restore` and `DELETE /api/v1/project/:id`. The archived-project detail view exposes both: the frontend's `renderArchivedProjectDetail` gets "Restore"/"Delete Permanently" buttons (the latter behind a `confirm()`, navigating back to the project list on success), and the TUI mirrors them on the Project screen as `u` (restore) and `D` (permanent delete, behind a `y`-to-confirm prompt).

**Frontend** (`frontend/index.html`, single-file vanilla JS): dark "Dispatch Ticket" visual identity (amber accent, monospace chrome type for labels/badges/tabs vs. sans for content, dashed "perforation" dividers on headers/cards/home-tiles) driven entirely by CSS custom properties in `:root`, with `color-scheme: dark` set so native date/time/select popups follow suit. Seven full-screen sections — Home (landing page), Lists, Notes, Log, Todo, Projects, Project Detail — each an `.app-overlay` div toggled by `showView(view, fromPopState, extra)`, with exactly one `.active` at a time. Home is the default view; every other section has its own "Home"/"Back" button to get back. Lists' overlay wraps the original header+sidebar+panel markup unchanged; Todo defaults to its List tab (not Add), and its priority field is a 0–5 dropdown of descriptions rather than free text. Each todo card's Edit button (`todoEditTodo`) repopulates the Add-tab form and switches its submit to `PUT /api/v1/todo/:id`, preserving `completed`/`created_at`/`completed_at`/`printed_at` from the original item so editing never touches completion state (both backends delete-and-recreate subtasks on update, so the form's free-text subtask textarea round-trips safely); Project Detail renders the same `renderTodoCard` but with `editable:false`, since those cards come from a separate fetch rather than the `todoItems` array Edit looks up against. Every `showView()` call, and (within Lists on mobile) every transition between the item panel and list selection, pushes a `history` entry; a `popstate` listener replays the matching state, so the mouse back/forward buttons and mobile swipe-back/forward gestures navigate the app instead of leaving it. `showView`'s optional third `extra` object (e.g. `{projectId}`) rides along in the pushed history state for views that need more than just a view name — Project Detail is the one other consumer of this besides Lists' own `{panel, categoryId}` state. Each list category has two independent, per-category toggles reachable from its "☰" options menu — `has_checkboxes` and `has_quick_add` (show/hide the "★ Quick Add" pane of saved common items) — both persisted server-side (`lists` crate). The Quick Add pane's per-item delete (×) only appears/works while that pane's "Edit" toggle is on; it never triggers from plain hover. Every "name"-like text input (project/group/category/item/note/log/todo titles, notebook names) carries a `lc-first` class handled by one delegated `input` listener that forces the first character lowercase live as the user types (`lowercaseFirstChar()`); the one prompt()-based rename flow applies the same helper once to its result instead of live. Todo/Lists/Notes/Log each have a client-side "Show Project Items" toggle, living in that section's "☰" view-options menu (a `.cat-menu-wrap`/`.cat-menu-dropdown` popover, the same pattern as the List-options menu) rather than a filter pill — project-associated items are hidden by default and only shown once switched on, filtering client-side without any new backend query param. Lists refreshes `projectListGroupIds` on every visit (`listsOnOpen`) since a project's group can be created elsewhere in the app without this view reloading, and a project-owned group's delete (×) button never renders (`deleteGroup` also refuses server-agnostic if somehow called on one anyway). Notes' menu additionally drops project notebooks (named after their slug) from the notebook filter pill row (`renderNotebookFilters`) and always excludes the shared `archive` notebook from both the pill row and the unfiltered "All" browse view — `archive` is reachable only via a deliberate "☰ → Archive" action (`notesOpenArchive`), which resets back to normal the next time Notes is opened rather than silently sticking. Their Add forms each carry a project `<select>` (populated from `GET /api/v1/project` via `ensureProjectsLoaded()`/`populateProjectSelect()`) that sets `project_title` (todo) or injects the project's `project-<slug>` tag (notes/log) on submit; picking a project in the Notes form also auto-fills (but doesn't lock) the notebook field to that project's slug. The Projects overlay lists/creates projects; selecting one navigates (via `showView('project-detail', false, {projectId})`) to the separate Project Detail overlay, which aggregates that project's todos (completed ones filtered out client-side)/notes/log/lists with a "+ Add" button per section (jumps to that overlay's Add tab with the project pre-selected via a one-shot `pendingProjectContext` var) and clickable list categories (drills into the main Lists overlay via the existing `selectGroup`/`selectCategory`); its own "☰" menu (rather than an always-visible header button) holds the "Archive" action (`archiveProject(currentProjectDetailId)`), going static-metadata-only once archived. Every section's top nav bar (and the Lists header's button row) splits its horizontally-scrollable buttons into an `.hbar-scroll` wrapper kept as a *sibling* of any `.cat-menu-wrap`, never a parent — a dropdown is absolutely positioned and gets clipped by whichever ancestor has non-`visible` `overflow`, not just hidden behind `z-index`, so the "☰" trigger has to live outside the scrolling box; tab labels were also shortened (`+ New` instead of `+ Quick Add`/`+ Add Entry`/`+ Add Todo`, `Clear` instead of `Clear checked`) to leave more room before that scrolling is even needed. Since some browsers/foldables reload the page outright on a fold/unfold screen-size-class change, `showView()` and each section's tab-switch persist `{view, extra}`/the active tab to `sessionStorage`, and the three Add forms (`npanel-add`/`lpanel-add`/`tpanel-add`) debounce-persist their field values there too (`persistFormState`/`clearFormState`); `restoreFormState()`/`restoreLastView()` replay both on boot, so a forced reload resumes where it left off instead of dropping back to Home — scoped to the same tab session only, and cleared on successful submit so a stale draft never resurfaces. The TUI (`tui/src/ui.rs`) mirrors the two-pane list/detail structure (`Screen::Project`) and the `Shift+P` hide-toggle on Todo/Notes/Log/Lists folded into each screen's existing selection-filter pattern (see `todo_hide_completed`/`visible_todo_indices`), though it does not yet mirror the project-scoped add dropdowns.

**Deployment**: dev and prod are the same machine — no Docker, no cross-compilation. `deploy.sh` builds `app` natively (`cargo build --release -p app`), installs it to `/usr/local/bin/manage_dan`, and installs/restarts a `manage_dan` systemd unit (`WorkingDirectory` = project root, so relative paths like `app.sqlite` and `data/logs/app.log` resolve the same as under `cargo run`) plus an nginx reverse proxy (serves `frontend/index.html`, proxies `/api/` and `/todo/` to `127.0.0.1:8080`). See the script's header comments for one-time setup (nginx, `nb`, `zip` for project archiving, `plugdev` group for USB printer access). For frontend-only changes (no backend/API/schema change), `deploy-frontend.sh` just re-copies `frontend/index.html` to the nginx static path — no rebuild, no service restart, so the running app's in-memory/monitor state is undisturbed.
